class Solution {

    Deque<Integer> stk = new ArrayDeque<>();

    void findL(int[] arr, int[] prevL, int[] nextL) {

        int n = arr.length;

  

        stk.clear();

  

        for (int i = 0; i < n; i++) {

            while (!stk.isEmpty() && arr[stk.peekLast()] > arr[i]) {

                nextL[stk.removeLast()] = i;

            }

  

            if (!stk.isEmpty()) {

                prevL[i] = stk.peekLast();

            }

  

            stk.addLast(i);

        }

    }

        Arrays.fill(prevL, -1);

        }

# 🟢 PART 1 — Abstraction & Thinking Layer

---

## 1️⃣ Mental Model vs Memorization

From your draft:

- You wrote `findL`
    
- You computed `prevL` and `nextL`
    
- Then you directly tried:
    

```
heights[i] * (nextL[i] - prevL[i])
```

That instinct makes sense.

You were thinking:

> “Width is the gap between the next smaller and previous smaller.”

That’s actually very close to correct.

But the subtle break happens here:

You included the blockers inside the width.

That’s the small conceptual gap.

---

### What Your Brain Was Doing

You treated:

```
nextL[i] - prevL[i]
```

as the number of bars available.

But those two indices are the **blockers**.

They are not part of the rectangle.

That’s why the width was off by 1.

The rectangle is not from `prevL` to `nextL`.

It is from:

```
(prevL + 1) to (nextL - 1)
```

That tiny geometric correction changes everything.

---

### The Shift

Instead of thinking:

> “Distance between boundaries”

Think:

> “How many bars are strictly between the first smaller on left and first smaller on right?”

That mental rephrasing makes the `-1` obvious.

Because you are excluding both blockers.

Once you see it as _excluding blockers_,  
the formula stops feeling magical.

---

## 🔵 2️⃣ Problem Classification (Conceptual, Not Tool-Based)

It’s tempting to classify this as:

- “Monotonic stack problem”
    
- “Next smaller element problem”
    

And you did exactly that with `findL`.

But that label hides the real invariant.

This is not about next smaller.

It’s about:

> How far can a bar extend while remaining the minimum?

Next smaller is just a tool to find where that extension stops.

If you think “stack problem,”  
you focus on mechanics.

If you think “extension until blocked,”  
you focus on geometry.

That shift removes confusion.

---

## 🟣 3️⃣ Design Decisions and Their Necessity

### Why `prevL` is initialized to `-1`

You asked:

> “Why `-1` needed?”

Because when no smaller exists on the left,  
the bar can extend all the way to index 0.

If `prevL[i] = -1`, then:

```
width = nextL[i] - (-1) - 1
```

Which correctly counts from index 0.

Without `-1`,  
you’d need special-case handling.

So `-1` is not arbitrary.

It encodes:

> “There is no blocker on the left.”

Same for `nextL` initialized to `n`.

It encodes:

> “There is no blocker on the right.”

Those sentinel values are geometric placeholders.

They represent imaginary walls outside the array.

---

### Why subtract `-1`

Your first attempt:

```
nextL[i] - prevL[i]
```

Felt reasonable.

But it quietly includes the blockers.

Subtracting 1 is not adjustment.

It is exclusion.

You are removing the two walls from the width count.

---

## 🟡 4️⃣ Reasoning Artifacts — Why Your Arrays Matter

Your decision to compute `prevL` and `nextL` explicitly:

```
int[] prevL
int[] nextL
```

Prevents a major confusion:

- You separate “finding boundaries”
    
- From “calculating area”
    

That separation makes the geometry visible.

It keeps you from mixing:

- Stack mechanics
    
- Width math
    
- Area calculation
    

All in one loop.

That clarity is not accidental.

It prevents the exact “voodoo feeling” you described earlier.

## 🟣 5️⃣ Algorithm Walkthrough (With Intent)

Inside `findL`:

```java
while (!stk.isEmpty() && arr[stk.peekLast()] > arr[i]) {
    nextL[stk.removeLast()] = i;
}
```

This line means:

> “If current bar is smaller,  
> then all taller bars just found their right boundary.”

That’s not just stack logic.

That is boundary discovery.

Then:

```java
if (!stk.isEmpty()) {
    prevL[i] = stk.peekLast();
}
```

This says:

> “The nearest smaller to my left is whoever is still on stack.”

So stack always stores indices of increasing heights.

That invariant ensures correctness.

## 🔵 8️⃣ Constraints & Tradeoffs (Reality-Aware)

The real constraint shaping this solution is:

> Every bar must be processed in O(1) amortized time.

Without monotonic stack,  
finding boundaries per bar would be O(n).

Which makes total O(n²).

Your structure respects that constraint.

The `-1` and `n` sentinel values remove special cases,  
keeping the math uniform.

That’s important for clarity and correctness.

## 🧩 9️⃣ Pattern Extraction (Reusable Thinking)

This is:

**“Nearest smaller boundary + range multiplication” pattern**

Plain language:

> When something expands until blocked,  
> find the blockers.  
> Count what lies strictly between them.

This applies to:

- Largest Rectangle in Histogram
    
- Sum of Subarray Minimums
    

It does not apply when:

- Expansion is not limited by a simple nearest constraint
    
- Or when ordering isn’t monotonic
    

Mental note:

> When width depends on “how far until smaller,”  
> stop scanning.  
> Find boundaries once.  
> Multiply the gap.
