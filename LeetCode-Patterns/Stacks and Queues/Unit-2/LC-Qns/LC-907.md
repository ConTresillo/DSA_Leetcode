class Solution {

    Deque<Integer> stk = new ArrayDeque<>();
    static final int MOD = 1000000007;

    void findL(int[] arr, int[] prevL, int[] nextL) {
        int n = arr.length;

        stk.clear();

        for (int i = 0; i < n; i++) {
            while (!stk.isEmpty() && arr[stk.peekLast()] > arr[i]) {
                nextL[stk.removeLast()] = i;
            }

            if (!stk.isEmpty()) {
                prevL[i] = stk.peekLast();
            }

            stk.addLast(i);
        }
    }

    public int sumSubarrayMins(int[] arr) {
        int n = arr.length;

        int[] prevL = new int[n];
        int[] nextL = new int[n];

        Arrays.fill(prevL, -1);
        Arrays.fill(nextL, n);  // IMPORTANT: default next = n

        findL(arr, prevL, nextL);

        long sum = 0;   // MUST be long

        for (int i = 0; i < n; i++) {
            long left = i - prevL[i];
            long right = nextL[i] - i;

            long contribution = (arr[i] * left % MOD) * right % MOD;

            sum = (sum + contribution) % MOD;
        }

        return (int) sum;
    }
}

# 🟢 PART 1 — Abstraction & Thinking Layer

---

## 1️⃣ Mental Model vs Memorization

You were actually thinking correctly.

Your comments show this progression:

- “If you pop then you found next lesser”
    
- “If you finally stop then you found prev lesser”
    
- “Should had paid attention to Mod thingy”
    
- “Wtf even with mod it doesnt work”
    

So your mental model was:

> Use stack → compute prev & next → apply formula → done.

That instinct is completely reasonable.  
And structurally, it’s correct.

Where things quietly break is not in the stack logic.  
It’s in what happens after.

The real shift here is:

You are no longer computing a structural property.  
You are computing a **large combinatorial count**.

And once multiplication enters:

```
(i - l) * (r - i) * arr[i]
```

you are no longer in “stack correctness land”.

You are in “integer overflow land”.

The stack never failed.

The numbers did.

---

## 🔵 2️⃣ Problem Classification (Conceptual, Not Tool-Based)

This problem looks like:

- Stack
    
- Range expansion
    
- Subarray counting
    

So it’s tempting to label it:

> “Monotonic stack problem.”

But the stack is not the core difficulty here.

The core invariant you were reasoning about is:

> Every element contributes based on how many subarrays it dominates.

That’s not a stack invariant.  
That’s a counting invariant.

The stack only gives boundaries.

The true structure of the problem is:

- Structural discovery (boundaries)
    
- Followed by combinatorial multiplication
    

That second phase is what actually shapes the solution.

And that’s where the constraint bites.

---

## 🟣 3️⃣ Design Decisions and Their Necessity

### Using a global stack

This makes clearing explicit.  
That’s fine. It keeps state visible.

It prevents accidental cross-contamination between calls.

---

### Assigning nextL during pop

This matches the exact invariant:

> When something smaller appears, it resolves next lesser.

That’s structurally correct.

---

### Filling `-1` for both prev and next

This makes boundary math clean later.

It prevents needing special cases during counting.

That’s good structure.

---

### Doing all multiplication in `int`

This is where the design quietly fights the problem.

You are multiplying:

- Up to ~30,000 values
    
- With ranges also up to ~30,000
    
- Then summing over ~30,000 elements
    

That easily exceeds 32-bit integer limits.

The structure is right.  
The numeric type is wrong.

---

## 🟡 4️⃣ Reasoning Artifacts — Why Your Comments Matter

> “If u pop then u found next Lesser”

This prevents a common confusion:

Accidentally assigning previous instead of next during pop.

You avoided that.

---

> “If u finally stop then u found prev Lesser”

This keeps the invariant clean:

After invalid elements are removed, the stack top is safe.

That prevents boundary leakage.

---

> “Should had paid attention to Mod thingy”

This shows awareness of constraints.

But mod alone doesn’t fix overflow.

Because overflow happens **before** the mod.

That’s the subtle trap.

---

> “Wtf even with mod it doesnt work”

This is the key moment.

The bug is not:

- Stack logic
    
- Boundary math
    
- Off-by-one
    

It is:

32-bit multiplication overflow corrupting intermediate results.

Once corrupted, `% 1000000007` can’t repair it.

That’s why you got negative output:

```
-786171066
```

That is classic signed 32-bit overflow.

Your mental model wasn’t incomplete.

It was numerically underspecified.

## 🟣 5️⃣ Algorithm Walkthrough (With Intent)

Inside `findL`:

- Pop while `arr[top] > arr[i]`
    
    - This correctly computes next lesser.
        
- After popping, top becomes previous lesser.
    
- Then you push `i`.
    

That logic is structurally correct.

The failure does not originate here.

---

In `sumSubarrayMins`:

```
sum += (i-l) * (r-i) * arr[i];
```

This is the critical line.

Even if final result fits inside mod range,  
the intermediate multiplication does not.

Example scale:

```

```

```

```

Overflow happens immediately.

So even before `%`, data is corrupted.

## 🧩 9️⃣ Pattern Extraction (Reusable Thinking)

Pattern:

> Boundary discovery + contribution counting

Use it when:

- Each element controls a dominance interval
    
- You count subarrays based on that interval
    

Does not apply when:

- Dominance is not monotonic
    
- Boundaries are dynamic or overlapping in non-monotonic ways
    

Mental note:

> When multiplication of distances appears, immediately check numeric bounds — not logic.

The stack was correct.

The arithmetic betrayed you.
